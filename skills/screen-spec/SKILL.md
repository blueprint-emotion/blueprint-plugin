# 화면명세 작성 가이드

화면명세는 `docs/screens/` 하위에 작성한다. 파일명이 `*screen*.md` 패턴과 일치하면 플랫폼에서 화면명세로 인식·노출한다.

### 참조 전략

이 스킬은 2단계 참조로 설계되어 있다.

1. **이 파일의 규칙과 포맷**만으로 대부분의 화면명세를 작성할 수 있다.
2. 완성된 형태가 궁금하면 `references/`를 참고한다.
   - `references/screen.md` — 페이지 (복잡한 화면 루트)
   - `references/area_image.md` — 영역 (기능 영역)
   - `references/area_option.md` — 영역 (기능 영역)

---

## 기능명세와의 역할 분리

- **기능명세** (rules): "이 도메인은 무엇을 가지고 있고, 어떤 규칙이 적용되는가"
- **화면명세** (elements): "이 화면에서 사용자가 무엇을 보고, 어떻게 표현되는가"

같은 도메인 속성이라도 화면마다 다르게 보인다. 그래서 **비즈니스 규칙은 기능명세**, **화면에 보이는 요소(elements)는 화면명세**에 둔다. 기능명세의 rules를 화면명세에 복사하지 않고 **링크로 참조**한다.

**elements에는 기능명세의 구체적 속성명(상품명, 판매가 등)을 직접 쓰지 않는다.** featureId로 어떤 기능 영역의 데이터인지만 명시하고, 표현 패턴(말줄임, 강조, 썸네일 등)을 기술한다. 화면에 정확히 어떤 속성이 보이는지는 와이어프레임과 기능명세의 조합으로 확인한다. 이렇게 하면 기능명세에서 속성명이 바뀌어도 화면명세를 수정할 필요가 없다.

---

## 파일 구조: 타입별 파일명

화면명세는 UI 타입별로 접두사를 다르게 둔다. 한 폴더에 여러 시트/다이얼로그/영역이 공존할 수 있다.

| 파일 패턴 | type | 설명 | title 접두사 |
|---|---|---|---|
| `screen.md` | `page` | 페이지 (라우트) | `화면_` |
| `sheet_{name}.md` | `sheet` | 시트 (우측 슬라이드) | `시트_` |
| `dialog_{name}.md` | `dialog` | 다이얼로그 (중앙 모달) | `다이얼로그_` |
| `area_{name}.md` | `panel` | 페이지의 레이아웃 영역/패널 | `영역_` |

- 페이지 1개/폴더가 원칙. 시트·다이얼로그·영역은 복수 가능
- frontmatter `type`은 파일명과 일치시킨다

### 분할 기준: 기능 영역 2개 이상이면 분할

| 기능 영역 수 | 구조 | 예시 |
|:---:|------|------|
| 1개 | `screen.md` 하나에 인라인 | 로그인, 에러 페이지 |
| 2개 이상 | `screen.md` + `area_*.md` | 상품 상세, 결제, 마이페이지 |

**파일 1개** — 기능 영역이 1개인 화면은 `screen.md` 하나에 유저스토리·elements·인수조건을 인라인으로 작성한다. 인터랙션·예외 케이스도 필요하면 같은 파일에 인라인.

**`screen.md` + 영역 파일** — 기능 영역이 2개 이상이면 루트에서 전체 레이아웃을 잡고, 각 영역을 `area_*.md`로 분리한다.

```
docs/screens/상품/product-detail/
├── screen.md                    ← 루트: 전체 레이아웃 + 영역 링크
├── area_image.md                ← 영역: 이미지 갤러리
├── area_info.md                 ← 영역: 상품 기본 정보
├── area_option.md               ← 영역: 옵션 선택
├── sheet_review.md              ← 시트: 리뷰 상세
├── dialog_zoom.md               ← 다이얼로그: 이미지 확대
├── wireframe.html               ← 와이어프레임
└── wireframe_mobile.html        ← 모바일 와이어프레임
```

이렇게 나누면:
- 에이전트가 특정 영역만 수정할 때 해당 파일만 읽으면 된다
- 기획자가 "옵션 영역 다시 기획해줘"라고 하면 파일 하나로 대응된다
- 각 영역 파일이 SSOT — 같은 내용을 중복 작성하지 않는다

## 루트 화면명세

화면 전체의 레이아웃과 상태를 정의한다. 루트의 유저스토리는 **화면 전체의 큰 목표**만 담고, 영역별 세부 시나리오는 각 영역 파일에 둔다.

### 프론트매터

```yaml
---
screenId: PRODUCT-DETAIL
title: 화면_상품 상세
type: page
purpose: 구매자가 상품 정보를 확인하고 구매를 결정하는 화면
viewport: [pc, mobile]
features: [PRODUCT, QNA]
---
```

| 필드 | 필수 | 설명 |
|------|:---:|------|
| `screenId` | Y | 화면 식별자. 대문자+하이픈 |
| `title` | Y | 타입 접두사 + 한글명 (e.g. `화면_로그인`, `시트_설정`, `다이얼로그_생성`) |
| `type` | Y | `page` / `sheet` / `dialog` / `panel` — 파일명 패턴과 일치 |
| `purpose` | Y | 이 화면이 존재하는 이유 (한 줄) |
| `viewport` | Y | 지원 뷰포트. `[pc]`, `[mobile]`, `[pc, mobile]` |
| `features` | Y | 참조하는 기능명세 도메인 목록 (대문자) |
| `group` | N | 그룹 폴더명. 없으면 미분류 |

`features`는 **도메인명**이며 `docs/features/{DOMAIN}.md`와 매칭된다.

`title` 접두사는 `type`과 일치시킨다 — `page` → `화면_`, `sheet` → `시트_`, `dialog` → `다이얼로그_`, `panel` → `영역_`.

frontmatter 직후 `# {접두사 없는 한글명}` h1을 둔다. h1은 본문 제목이라 접두사 없이 짧게.

```markdown
---
screenId: LOGIN
title: 화면_로그인
type: page
...
---

# 로그인
```

### 본문

| 섹션 | 필수 | 역할 |
|------|:---:|------|
| `## 유저스토리` | 조건부 | 화면 전체의 큰 목표 1~2개. **기능 영역 2개 이상(area_*.md 분할)**이면 선택(N), **단일 영역 화면**이면 필수(Y) — 영역 파일이 없어 인라인으로 작성해야 하므로. 각 항목에 **우선순위(Must/Should/Could)** 표기 |
| `## 화면 구성` | Y | 레이아웃 + 영역 파일 링크 (또는 인라인 elements) |
| `## 상태` | Y | 화면 레벨 상태 (로딩, 정상, 빈 상태, 에러 UI) |
| `## 인터랙션` | N | 모바일 제스처·키보드 동작 — `구성요소 / 인터랙션 / 상세` 표. 탭·스와이프·스크롤·자동 포커스·롱프레스 등. 와이어프레임 Web Component에 드러나지 않는 행동 규칙을 명시 |
| `## 예외 케이스` | N | 외부 시스템·환경 실패 케이스 — `상황 / 처리 방식` 표. API 오류, 네트워크 오프라인, 비로그인 접근, 권한 없음, 영업시간 외 등 **실제 발생 가능한** 상황. `## 상태`(UI 상태)와 구분: 상태는 화면이 보여주는 뷰, 예외는 외부 요인으로 인한 이탈 처리 |
| `## 인수조건` | N | 영역 간/화면 간 규칙. 영역별 인수조건은 각 영역 파일에 둔다 |
| `## 진입 경로` | N | 이 화면에 도달하는 경로 |
| `## 비즈니스 규칙` | N | 기능명세에 없는 화면 고유 규칙 |
| `## 비고` | N | 리뷰 시 알아둘 점 — 화면 단위 고유 정보. 와이어프레임의 `bp-description-note`로 그대로 옮겨진다 (유일한 출처) |

> **데이터 갱신 주기는 여기에 쓰지 않는다.** "이 화면이 실시간 반영되는지, 새로고침해야 갱신되는지"는 **도메인 속성**이므로 [feature-spec의 `## 데이터 생명주기` 섹션](../feature-spec/SKILL.md)에 둔다. 화면명세는 `features[]` 링크로 자동 연결되므로 중복 서술 불필요. 단, 외부 시스템 실패(API 오류·오프라인)에 대한 화면별 대응은 `## 예외 케이스`에 작성한다.

### 화면 구성 작성법

**복잡한 화면** — 각 영역을 파일 링크로 참조:

```markdown
## 화면 구성

상단은 2컬럼 그리드(1:1)로 구성한다.

- **좌측**: [이미지 갤러리](./area_image.md)
- **우측 상단**: [상품 기본 정보](./area_info.md)
- **우측 하단**: [옵션 선택](./area_option.md)

하단은 탭 영역으로, "상세정보"와 "Q&A" 탭을 전환한다.
```

**단순한 화면** — 영역과 elements를 인라인으로:

```markdown
## 화면 구성

### [로그인 폼](../../features/AUTH.md#LOGIN)

**elements**
- AUTH__LOGIN 인증 정보를 폼으로 입력 (보안 항목 마스킹 + 보기 토글)
- 로그인 버튼
- 소셜 로그인 버튼 (GitHub)
```

### 유저스토리 레벨 분담

루트와 영역에 같은 유저스토리를 중복 작성하지 않는다. 스케일이 다르다.

| 레벨 | 관점 | 템플릿 |
|---|---|---|
| 루트 `screen.md` | 화면 전체의 큰 목표 — 사용자가 이 화면에 왜 왔는가 | "~로서, **이 화면을 통해** ~하고 싶다" |
| 영역 `area_*.md` | 영역의 세부 시나리오 — 이 영역에서 무엇을 하고 싶은가 | "~로서, **이 영역에서** ~하고 싶다" |

예:

```markdown
<!-- screen.md (상품 상세) -->
## 유저스토리
- 구매자로서, 상품 정보를 종합적으로 확인하고 구매 결정을 내리고 싶다
```

```markdown
<!-- area_image.md -->
## 유저스토리
- 구매자로서, 상품을 여러 각도에서 확대해 살펴보고 싶다 — 실제 품질을 짐작하기 위해
```

```markdown
<!-- area_option.md -->
## 유저스토리
- 구매자로서, 색상·사이즈를 고르고 실시간 재고를 확인하고 싶다
```

루트는 1~2개로 제한. 구체적 시나리오는 영역 파일에서.

### 유저스토리 우선순위

각 유저스토리 **앞에** 우선순위 라벨을 붙인다. 백로그 정렬·grep 용이성·릴리스 범위 협상에 쓰인다.

| 라벨 | 의미 |
|---|---|
| `[Must]` | 이 화면의 존재 이유. 빠지면 화면이 성립하지 않음 |
| `[Should]` | 있어야 완성도 있는 경험. 초기 릴리스에 포함 권장 |
| `[Could]` | 있으면 좋지만 없어도 무관. 후속 개선 후보 |

예:
```markdown
## 유저스토리
- [Must] 구매자로서, 이 영역에서 옵션을 선택하고 재고를 실시간으로 확인하고 싶다
- [Should] 구매자로서, 품절 옵션이 시각적으로 구분되길 원한다 — 헛된 시도를 피하기 위해
- [Could] 구매자로서, 옵션 조합별 이미지 미리보기를 보고 싶다
```

### 인터랙션 섹션 작성법

와이어프레임 Web Component가 외형을 보여주지만, **제스처·키보드·자동 동작** 같은 행동 규칙은 코드만으로 드러나지 않는다. 다음 패턴이 있으면 `## 인터랙션`을 작성한다.

| 구성요소 | 인터랙션 | 상세 |
|---|---|---|
| 이미지 갤러리 | 스와이프 | 좌우로 드래그하여 이전/다음 이미지 전환 |
| 옵션 선택기 | 자동 포커스 | 화면 진입 시 첫 번째 선택지에 포커스 |
| 탭 헤더 | 스크롤 | 가로 스크롤로 가려진 탭 접근 |
| 버튼 | 길게 누르기 | 500ms 이상 누르면 컨텍스트 메뉴 표시 |

표 외 서술 대신 표만 유지한다. "기획 의도"가 아닌 **동작 기준** 위주.

### 예외 케이스 섹션 작성법

`## 상태`는 **화면이 정상적으로 보여주는 UI 상태**(로딩·빈 상태·에러 뷰)를 다루고, `## 예외 케이스`는 **외부 시스템·환경 요인으로 인한 이탈 처리**를 다룬다. 둘은 역할이 다르다.

| 상황 | 처리 방식 |
|---|---|
| API 500 응답 | "일시적 오류" 토스트 노출 + 재시도 버튼 |
| 네트워크 오프라인 | 오프라인 배너 + 캐시된 마지막 데이터 유지 |
| 비로그인 접근 | 로그인 화면으로 리다이렉트, 현재 경로를 쿼리로 보존 |
| 권한 없음 (403) | 권한 없음 화면(`forbidden`)으로 이동 |
| 영업시간 외 | 안내 배너 + 버튼 비활성화 |

"에러 처리" 한 줄로 퉁치지 않는다. 실제 발생 가능한 상황별로 분리.

## 영역 화면명세

화면의 특정 영역(이미지 갤러리, 옵션 선택 등)을 독립 문서로 기술한다. 파일명은 `area_{name}.md`.

### 프론트매터

```yaml
---
title: 영역_옵션 선택
type: panel
features:
  - id: PRODUCT__OPTION
    ref: ../../features/PRODUCT.md#OPTION
  - id: PRODUCT__PRICE
    ref: ../../features/PRODUCT.md#PRICE
---
```

| 필드 | 필수 | 설명 |
|------|:---:|------|
| `title` | Y | 영역 한글명 |
| `features` | **Y (1개 이상 필수)** | 이 영역이 **UI로 표현하는** 기능 목록. 각 항목은 `id`(featureId) + `ref`(기능명세 링크). **빈 배열(`[]`) 금지** |

각 `features[].id`는 기능명세의 특정 섹션과 매칭된다. `ref`는 해당 섹션의 상대 경로. 플랫폼이 이 값으로 rules를 자동 로드하고, 와이어프레임 생성 시 에이전트가 `ref`를 따라가 구체적 속성을 확인한다.

**`features[]`에 넣는 기준 — UI 구분 단위만**

기능명세에 있다고 다 넣는 게 아니다. **이 영역이 실제로 UI 영역으로 표현하는 기능**만 넣는다. `features[]` 항목은 와이어프레임의 `<bp-section data-feature="...">`와 1:1 대응하기 때문이다.

| 이 영역이 그 기능을… | features[] 포함? |
|---|:---:|
| UI 영역(시각적으로 구분되는 블록)으로 표현 | ✅ YES |
| 맥락·rule 참조만 하고 별도 UI 없음 (내부 동작·URL 동작·파일 규약 등) | ❌ NO — description 본문 링크로 참조 |

예: 뷰어 영역은 iframe 렌더링(`WIREFRAME__RENDERING`)과 핀 오버레이(`COMMENT__PIN`)를 UI로 구분한다 → `features[]`에 2개. 브릿지 스크립트(`WIREFRAME__BRIDGE`)는 내부 동작이라 별도 UI 없음 → `features[]`에 넣지 않고 iframe 설명 본문에 `[브릿지](../../../features/WIREFRAME.md#bridge-...)` 링크로 참조.

**왜 빈 배열이 금지인가**: `features`는 영역과 기능명세를 잇는 유일한 구조적 링크다. 빈 배열로 두면 기능명세에서 toc 항목이 삭제되어도 영역 파일이 그 잔재(elements·인수조건의 UI 언급)를 추적할 수 없다. 영역에 UI로 구분할 feature가 하나도 없다면 그 파일은 영역(panel)이 아니라 화면(page)의 chrome일 가능성이 높다 — 분리하지 말고 `screen.md`에 인라인으로 옮길 것.

영역 파일의 `title`은 **`영역_` 접두사 + 한글명**. frontmatter 직후 `# {접두사 없는 한글명}` h1을 둔다.

### 본문

| 섹션 | 필수 | 역할 |
|------|:---:|------|
| `## 유저스토리` | Y | 이 영역의 사용자 시나리오. "~로서, 이 영역에서 ~하고 싶다, ~하기 위해". 각 항목에 **우선순위(Must/Should/Could)** 표기 |
| `**elements**` (bold 라벨, `##` 아님) | Y | 이 영역에서 사용자가 보는 요소. `## 유저스토리` 다음, 다른 `##` 섹션 앞에 위치 |
| `## 인터랙션` | N | 이 영역 고유의 제스처·키보드 동작 — `구성요소 / 인터랙션 / 상세` 표 |
| `## 예외 케이스` | N | 이 영역에서 발생 가능한 외부 실패 — `상황 / 처리 방식` 표. 영역 단위 API 실패·빈 데이터·권한 거부 등 |
| `## 인수조건` | Y | 이 영역의 완성 기준 체크리스트 |
| `## 비고` | N | 이 화면 리뷰 시 알아둘 점 — 기능명세의 rule이 아닌 **화면 단위 고유 정보**(엣지 케이스, 주의사항, 구현 제약). **와이어프레임의 `bp-description-note`로 그대로 옮겨진다** (유일한 출처) |

### elements 작성법

**형식**: `**elements**` 는 `## 헤딩`이 아니라 **bold 인라인 라벨**이다. 바로 아래에 블릿 리스트로 요소를 나열한다.

| 위치 | 형식 |
|---|---|
| 영역 파일 (`area_*.md`) | `## 유저스토리` 다음, 다른 `## 섹션`(인터랙션 등) **앞**에 둔다 |
| 단순 화면 인라인 (`screen.md` 단일 영역) | `## 화면 구성` 안의 `### [영역명](feature link)` 헤딩 **직후**에 둔다 |

**작성 원칙**: 기능명세의 구체적 속성명(상품명, 판매가 등)을 나열하지 않는다. **featureId로 영역을 명시**하고, 그 영역의 데이터를 **어떤 표현 패턴으로** 보여주는지를 기술한다.

**작성 원칙:**
1. featureId를 먼저 쓰고, 해당 영역의 데이터를 어떤 UI 패턴으로 표현하는지 기술한다
2. 표현 패턴 유형별로 묶는다 (텍스트 항목, 가격 항목, 이미지 항목 등)
3. 기본 표현(일반 텍스트 표시, 기본 텍스트 입력 등)은 생략한다. 특수한 패턴만 명시한다
4. 화면 고유의 조건부 표현 (품절 뱃지, 로딩 스켈레톤 등)은 별도로 기술한다
5. featureId와 무관한 UI 요소 (버튼, 네비게이션 등)는 그대로 쓴다

**잘못된 예** — 속성명 직접 사용:

~~~~markdown
**elements**
- 상품명 (2줄 말줄임)
- 판매가 (강조)
- 대표 이미지 (썸네일)
- 재고 표시 (품절 시 "품절" 뱃지)
~~~~

**올바른 예** — featureId 참조 + 표현 패턴:

~~~~markdown
**elements**
- PRODUCT__INFO 속성을 카드 형태로 표시
  - 텍스트 항목: 2줄 말줄임
  - 가격 항목: 강조 표시
- PRODUCT__IMAGE 대표 이미지를 썸네일로 표시
- 품절 상태일 때 "품절" 뱃지 표시
~~~~

**표현 패턴 유형:**

| 유형 | 설명 | 예시 |
|------|------|------|
| 텍스트 항목 | 도메인의 텍스트성 속성 | 2줄 말줄임, 전체 표시, 1줄 말줄임 |
| 가격 항목 | 가격·금액성 속성 | 강조 표시, 취소선, 할인율 병기 |
| 이미지 항목 | 이미지성 속성 | 썸네일, 갤러리, 확대 보기 |
| 상태 항목 | 상태값 속성 | 뱃지, 색상 구분, 아이콘 |
| 액션 항목 | 사용자 행동 요소 | 버튼, 링크, 토글 |

## 디렉토리 구조

```
docs/screens/
├── {그룹}/              ← 그룹 폴더 (한글 가능)
│   └── {화면폴더}/      ← 화면 폴더
│       ├── screen.md       ← 페이지 (루트)
│       ├── area_*.md       ← 영역/패널 (복수 가능)
│       ├── sheet_*.md      ← 시트 (복수 가능)
│       ├── dialog_*.md     ← 다이얼로그 (복수 가능)
│       └── *.html          ← 와이어프레임
│
├── {화면폴더}/          ← 그룹 없이 → "미분류"
```

### 플랫폼 렌더링 규칙

- `docs/screens/**/*.md` → 마크다운 렌더링 (화면명세). 파일명 패턴 무관
- `.html` → iframe 렌더링 (와이어프레임)
- html 파일명에 `mobile` 포함 → 370px 뷰포트, 그 외 → 100% 너비
- 화면폴더명은 한 번 정하면 변경하지 않는다. 기존 코멘트 연결이 끊어지기 때문이다.

## 자주 하는 실수

| 실수 | 왜 문제인가 |
|------|------------|
| 기능명세의 rules를 화면명세에 복사 | 두 곳에서 관리하면 불일치가 생긴다. feature 링크로 자동 연결된다 |
| 루트 features에 featureId를 나열 | 루트에는 도메인명(`PRODUCT`), 영역에는 `features[].id`로 featureId + ref |
| 유저스토리나 인수조건 생략 | 화면의 존재 이유와 완성 기준이 없으면 개발자가 판단할 수 없다 |
| 루트와 영역에 같은 유저스토리 중복 작성 | 스케일이 다르다 — 루트는 화면 전체의 큰 목표, 영역은 그 영역 고유 시나리오. 같은 문장을 두 곳에 쓰면 SSOT가 깨진다 |
| 단일 영역 screen.md에 유저스토리 생략 | 영역 파일이 없는 화면(e.g. 로그인, 에러 페이지)은 screen.md가 유저스토리의 유일한 출처. `purpose` 한 줄만으로는 완성 기준을 세울 수 없다 |
| 유저스토리에 우선순위 누락 | Must/Should/Could가 없으면 릴리스 범위 협상·백로그 정렬이 불가능하다. 특히 Could 범위를 Must로 오해하면 범위가 무한히 늘어난다 |
| `## 상태`에 예외 케이스를 섞어 씀 | 상태는 정상 흐름 안의 UI 뷰(로딩·빈 상태·에러 뷰)만, 예외는 외부 시스템·환경 실패(API/네트워크/권한)만. 섞이면 QA·테스트 케이스 작성 시 커버리지 추적이 끊긴다 |
| 인터랙션을 와이어프레임만 보고 유추하라고 생략 | 제스처·자동 포커스·키보드 동작은 bp-* 컴포넌트 외형만으로 드러나지 않는다. 명시하지 않으면 모바일 구현이 누락된다 |
| 복잡한 화면을 파일 하나에 전부 작성 | 에이전트가 전체를 읽어야 하고 기획자도 부담이 된다. 영역별로 분할한다 |
| 같은 내용을 여러 screen 파일에 중복 | 각 파일은 SSOT. 필요하면 링크로 참조한다 |
| elements에 기능명세 속성명을 직접 나열 | 기능명세에서 속성명이 바뀌면 모든 화면명세를 수정해야 한다. featureId 참조 + 표현 패턴으로 쓴다 |
| `area_*.md`의 `features: []` (빈 배열) | 영역과 기능명세 간 유일한 구조적 링크가 끊긴다. 기능명세 변경의 영향을 추적할 수 없어 잔재가 남는다 |
| 기능명세에 있는 모든 feature를 `features[]`에 다 넣기 | `features[]`는 UI 구분 단위만. 맥락 rule 참조용이면 description 본문에서 마크다운 링크로 (와이어프레임 bp-section과 1:1 안 맞으면 lint가 실패하거나 UI 없는 feature를 억지로 구조화하게 됨) |
| AC가 기능명세 rule을 그대로 베끼기 | rule은 SSOT(기능명세)에만. AC는 영역 단위의 화면적 완성 기준만 — "[기능명세](#anchor) 참조"로 링크 |
| `## 비고`에 기능명세 rule을 복붙 | rule은 기능명세가 SSOT. 비고는 화면 리뷰용 고유 정보(엣지 케이스·주의사항·구현 제약)만. 비고 내용은 와이어프레임 `bp-description-note`에 그대로 렌더되므로, rule을 넣으면 SSOT가 2중(기능명세·비고·와이어프레임)으로 퍼진다 |
| 화면 폴더명 변경 | 기존 코멘트 연결이 끊어져 복구할 수 없다 |
