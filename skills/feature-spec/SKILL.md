# 기능명세 작성 가이드

기능명세는 `docs/features/`에 **도메인별 1파일**로 작성한다. 이 파일이 해당 도메인의 **유일한 정보 소스(SSOT)**.

### 참조 전략

이 스킬은 2단계 참조로 설계되어 있다.

1. **이 파일의 규칙과 포맷**만으로 대부분의 기능명세를 작성할 수 있다.
2. 완성된 형태가 궁금하면 `references/PRODUCT.md`를 참고한다. 전체를 읽지 말고 필요한 섹션만 확인한다.

---

## 도메인을 서술한다, 화면을 서술하지 않는다

기능명세의 핵심은 **"이것은 무엇인가"**를 정의하는 것이다. "이 화면에서 어떻게 보이는가"가 아니다.

하나의 도메인 기능은 여러 화면에서 쓰인다:

- 상품 목록 → PRODUCT__INFO + PRODUCT__IMAGE + PRODUCT__PRICE
- 상품 상세 → PRODUCT 전체 + OPTION + PRICE
- 장바구니 → PRODUCT__INFO + PRODUCT__OPTION + PRODUCT__PRICE

기능명세가 특정 화면에 종속되면 다른 화면에서 재사용할 수 없다. 화면별 표현(레이아웃, 말줄임, 호버 등)은 화면명세와 와이어프레임이 담당한다.

**기능명세의 도메인 구조 선언("X는 Y, Z를 가진다")이 속성의 SSOT이다.** 화면명세에서는 속성명을 직접 언급하지 않고 featureId로 영역만 참조하므로, 속성의 추가·변경·삭제는 기능명세 한 곳에서만 관리하면 된다.

### 기획자의 언어로 쓴다

이 문서를 작성·검수하는 사람은 기획자다. `integer`, `varchar`, `PK`, `FK` 같은 DB 스키마 표기 대신 자연어로 "이것이 무엇인지"를 설명한다.

구조는 충분히 정돈한다. 개발자가 이 문서를 근거로 에이전트와 함께 DB 모델링을 수행하기 때문이다. 각 `##` 섹션의 rules에서 도메인 구조와 제약조건을 도출할 수 있을 정도면 충분하다.

### 화면에 보이는 것(elements)은 화면명세에 쓴다

사용자가 화면에서 보는 요소의 **표현 방식**은 화면명세의 `**elements**`에 작성한다. 단, 기능명세의 구체적 속성명(상품명, 판매가 등)을 직접 쓰지 않고 **featureId로 영역만 참조**하고 표현 패턴을 기술한다:

- 상품 목록: PRODUCT__INFO를 카드 형태로 (텍스트 2줄 말줄임, 가격 강조), PRODUCT__IMAGE를 썸네일로
- 상품 상세: PRODUCT__INFO를 전체 표시, PRODUCT__PRICE를 정가 취소선 + 판매가 강조
- 장바구니: PRODUCT__INFO를 1줄 요약, PRODUCT__OPTION 선택 내역 표시

기능명세에는 "상품은 이름, 판매가, 상태를 가진다"라는 **도메인 구조**와 "판매가는 정가를 초과할 수 없다"라는 **비즈니스 규칙**만 쓴다.

## 디렉토리 구조

```
docs/features/
├── INDEX.md        ← 전체 도메인 맵 (목차)
├── PRODUCT.md      ← 도메인: 상품
├── ORDER.md        ← 도메인: 주문
└── COMMON.md       ← 도메인: 공통
```

파일명 = 도메인명 (대문자).

---

## featureId

```
{도메인}__{기능}__{하위기능}__{...}
```

더블 언더스코어(`__`)로 계층을 구분하고, 단일 언더스코어(`_`)로 단어를 구분한다. 대문자만 사용한다. 와이어프레임 HTML의 `<bp-section data-feature="...">` 와 매칭된다.

```
PRODUCT__INFO                    ← 2 depth
PRODUCT__LIST__CARD              ← 3 depth
ORDER__CHECKOUT__PAYMENT         ← 3 depth
```

프론트매터 `toc`에는 도메인 접두사 없이 짧은 id만 쓴다 (`INFO`, `LIST`). 도메인은 이미 프론트매터 상단에 있으므로 반복할 이유가 없다. 전체 featureId가 필요할 때는 `{domain}__{id}`로 조합한다.

## 프론트매터

```yaml
---
domain: PRODUCT
title: 기능_상품
toc:
  - id: INFO
    label: 상품 기본 정보
  - id: IMAGE
    label: 상품 이미지
  - id: OPTION
    label: 상품 옵션
  - id: PRICE
    label: 가격
---

# PRODUCT (상품)
```

- `title`은 **`기능_` 접두사 + 한글명**. 플랫폼 규칙(옵시디언 그래프 구분).
- frontmatter 직후 `# {DOMAIN} ({한글명})` h1을 동반한다. h1은 접두사 없이 짧게.
- `toc` 배열의 각 항목은 본문의 `##` 헤딩과 1:1 대응한다. 하위 기능(`###` 이하)은 `toc`에 넣지 않는다. 각 항목의 `label`은 해당 섹션의 한글명.

에이전트는 이 `toc`만 읽고 파일의 구조를 파악한 뒤, 필요한 섹션으로 점프한다. 전체 파일을 읽지 않아도 되는 구조를 의도적으로 설계한 것이다.

## 본문 구조

### 헤딩 = 기능 계층

| depth | 헤딩 | 역할 | 예시 |
|:---:|:---:|------|------|
| 1 | `#` | 도메인 | `# PRODUCT (상품)` |
| 2 | `##` | 대기능 | `## INFO (상품 기본 정보)` |
| 3 | `###` | 하위 기능 | `### CARD (상품 카드)` |
| 4~5 | `####`~ | 세부 기능 | (필요시) |

대부분 2~3 depth면 충분하다. 최대 5.

각 기능 헤딩 아래에는 `**rules**`만 작성한다. 자식 기능이 있어도 본인의 rules를 가질 수 있다.

### rules 작성법

rules는 두 가지를 정의한다:

**1. 도메인 구조** — 이 개념이 무엇을 가지고 있는가

첫 번째 rule로 해당 개념의 구성 요소를 자연어로 선언한다. 이것이 개발자가 DB 스키마를 도출하는 근거가 된다.

```
- 상품은 이름, 설명, 판매자, 상태, 별점, 리뷰 수를 가진다
- 옵션은 단계, 상위 옵션, 그룹명, 옵션값, 추가금, 재고, 상태를 가진다
```

**2. 비즈니스 규칙** — 화면과 무관하게 항상 성립하는 제약조건

```
- 판매가는 정가를 초과할 수 없다
- 재고가 0이면 품절 상태가 된다
- 삭제된 상품은 어디에서도 조회되지 않는다
```

화면명세에 써야 할 것 (기능명세에 쓰지 않는다):
- "가격을 빨간색으로 표시한다" — UI 표현
- "호버 시 이미지가 확대된다" — 인터랙션
- "상품명을 2줄까지만 표시하고 말줄임한다" — 화면별 표현

### 기능명세는 도메인 전체, 화면명세는 그 화면이 쓰는 것만

기능명세는 **도메인의 모든 기능**을 담는다. 화면은 그중 **자신이 UI로 표현하는 것만** `features[]`에 담는다.

| 상황 | 올바른 선택 |
|------|-----------|
| 기능명세에 100개 feature가 정의되어 있음 | 화면명세 `features[]`에 100개 전부 넣는 것 ❌ |
| 이 화면이 실제로 UI 구분해서 보여주는 feature가 3개 | `features[]`에 3개만 넣는다 ✓ |
| 그 화면에서 맥락·rule 참조만 필요하고 UI 구분 없음 | `features[]`에 넣지 않음. 필요한 rule은 `bp-description-element` 본문에서 **마크다운 링크로 참조** ✓ |

**`features[]` 항목 = 와이어프레임의 `<bp-section>`과 1:1.** 스킬 규칙 "각 features[].id마다 별도 bp-section을 생성한다"를 유지하려면, `features[]`에 넣는 것은 실제 UI 구분 단위여야 한다. 내부 동작(브릿지 스크립트 등)·URL 동작(딥링크 등)·파일 생성 규약 같은 것들은 관련 UI 요소의 description 본문에서 링크로 참조할 뿐 `features[]`에는 넣지 않는다.

### rule은 기능명세 밖으로 복제되지 않는다

rule의 유일한 SSOT는 기능명세다. 다른 문서에서 rule 내용이 필요하면 **링크로 참조**한다.

| 대상 | 올바른 방식 |
|------|-----------|
| 화면명세의 `## 인수조건` | rule을 복붙하지 말고 "[기능명세#섹션](path#anchor) 참조"로 연결 |
| 화면명세의 `## 비고` | rule이 아닌 화면 리뷰용 고유 정보만. 비고는 와이어프레임 `bp-description-note`로 그대로 렌더되므로 rule이 들어가면 3곳(기능명세·비고·와이어프레임)으로 퍼진다 |
| 와이어프레임의 `bp-description-element` 본문 | 기능명세의 구체적 속성명·rule 문장을 쓰지 말고 **링크로 참조** |
| 와이어프레임의 `bp-description-note` | 화면명세의 `## 비고` 항목만 옮김. 기능명세 rule 금지 |

**도메인 구조 완성도 체크리스트:**

화면명세에서 속성명을 직접 쓰지 않으므로, 기능명세의 도메인 구조가 속성의 유일한 SSOT이다. 개발자가 이 rules만 읽고 모델링이 가능하도록 다음을 확인한다:

- [ ] 필수/선택 여부가 명시되어 있는가
- [ ] 기본값이 있는 속성에 기본값이 적혀 있는가
- [ ] 값의 범위 또는 선택지가 나열되어 있는가 (상태값 등)
- [ ] 다른 도메인과의 관계가 명시되어 있는가 (주문은 상품을 참조한다)

## 데이터 생명주기 (선택 섹션)

도메인 데이터가 **언제 어떻게 갱신되는가**를 도메인 언어로 기술한다. 기획자가 작성하고, 개발자가 이 내용을 근거로 `docs/DB.md`의 스키마·구독 전략을 작성·갱신한다.

### 작성 위치

`## 데이터 생명주기` — 본문의 모든 feature 섹션(`## INFO`, `## IMAGE` 등) **뒤에** 둔다.

이 섹션은 **`toc`에 넣지 않는다.** toc는 feature(= `<bp-section data-feature="...">`과 매칭되는 UI 구분 단위)만 담는 규약이고, 데이터 생명주기는 도메인 메타데이터다. featureId가 없으므로 toc에 넣으면 규약이 깨진다.

### 작성 원칙

1. **도메인 언어로** — "Postgres Changes 구독", "Redis TTL 300s" 같은 구현 용어 금지. "실시간 반영", "세션 동안 고정", "관리자 수정 시점만 변경" 같은 기획 언어.
2. **기본 정책 + 예외** 구조 — 도메인 전체의 기본 정책을 한 줄로, 개별 feature가 다르면 예외로 나열.
3. **개발자 힌트** — 마지막에 `> 구현 참고: [DB.md#{table}](../DB.md#{table})` 한 줄로 스키마 대응처를 링크. 이 링크는 개발자가 채우는 것이지 기획자의 필수사항은 아니다.

### 예시

```markdown
## 데이터 생명주기

**기본**: 사용자가 화면에 진입할 때마다 최신 상태를 조회한다. 변경 빈도 저빈도.

**예외**:
- **COMMENT**: 실시간 반영 — 다른 사용자의 코멘트 추가·수정·삭제가 현재 열린 화면에 즉시 나타난다
- **VERSION__TAG**: 변경 저빈도, 세션 동안 캐시 가능 — 태그는 생성 후 변경되지 않는다
- **PROJECT__ROLE**: 사용자 로그인 시점에 결정되며, 세션 동안 고정

> 구현 참고: [DB.md#comments](../DB.md#comments), [DB.md#versions](../DB.md#versions)
```

### 선택지 어휘 (권장)

일관성을 위해 다음 어휘 중에서 고른다:

| 표현 | 의미 |
|---|---|
| **실시간 반영** | 다른 사용자의 변경이 즉시 현재 화면에 나타남 (Realtime 구독) |
| **사용자 행동 시 갱신** | 새로고침·재진입·명시적 액션 시점에만 갱신 |
| **세션 동안 고정** | 로그인 세션 내에서는 변경되지 않음 (역할·권한 등) |
| **변경 불가 / 변경 저빈도** | 생성 후 불변 또는 드물게 변경 (태그·정산 기록 등) |
| **관리자 수정 시점만 변경** | 운영자 개입이 없으면 변경 없음 (공지·가격 등) |

### DB.md의 역할

기획자는 **"무엇이 어떻게 살아있는가"**만 쓴다. 개발자가 feature-spec의 데이터 생명주기를 읽고 `DB.md`에:
- 테이블 스키마
- Realtime 구독 채널 / Broadcast 여부
- 캐시 TTL / 무효화 규칙
- 인덱스·쿼리 전략

…을 작성한다. **DB.md는 feature-spec의 구현 반영물이다.** 기획자는 DB.md를 직접 편집하지 않는다.

## INDEX.md

각 도메인 파일의 `toc`를 취합한 전체 목차. 에이전트가 프로젝트의 기능 전체를 훑어볼 때 이 파일 하나만 읽으면 된다.

```markdown
# 기능 명세 목차

## PRODUCT (상품)
- INFO: 상품 기본 정보
- IMAGE: 상품 이미지
- OPTION: 상품 옵션
- PRICE: 가격

## ORDER (주문)
- CART: 장바구니
- CHECKOUT: 결제
```

## 자주 하는 실수

| 실수 | 왜 문제인가 |
|------|------------|
| 화면에 보이는 요소(elements)를 기능명세에 작성 | elements는 화면명세에 쓴다. 같은 속성도 화면마다 다르게 보이기 때문이다 |
| rules에 "빨간색", "말줄임", "호버" 같은 UI 표현 | 화면 표현은 화면명세가 담당한다. 기능명세는 비즈니스 규칙만 |
| rules에 `PK`, `FK`, `integer` 같은 DB 표기 | 기획자가 작성·검수하는 문서이므로 비즈니스 언어를 쓴다 |
| 도메인 구조 선언 없이 제약조건만 나열 | 첫 rule에서 "X는 Y, Z를 가진다"로 구성 요소를 선언해야 DB 모델링 근거가 된다 |
| toc id에 `PRODUCT__INFO`처럼 도메인 접두사 | 도메인은 프론트매터에 이미 있으므로 `INFO`면 충분 |
| featureId 계층을 단일 언더스코어로 구분 | 단어 구분(`_`)과 계층 구분(`__`)이 혼동된다 |
| toc 항목 삭제·이름 변경 시 featureId grep 미수행 | 화면명세(`area_*.md`의 `features[].id`)·와이어프레임(`<bp-section data-feature="...">`)·요약 문서(INDEX, ROUTING, screen.md 본문)에 잔재가 남는다 |
| rules가 화면명세의 `## 비고`나 와이어프레임의 `bp-description-note`에 복제되어 있음 | rule의 SSOT는 기능명세다. 화면 리뷰 중 rule이 필요하면 **기능명세 링크**로만 참조한다. 비고/note는 rule을 담지 않는다 (담으면 같은 규칙을 여러 곳에서 수정해야 함) |
| 데이터 생명주기에 "Postgres Changes", "Redis TTL" 같은 구현 용어 사용 | 기획자가 작성하는 문서이므로 도메인 언어(실시간 반영·세션 동안 고정 등)만 쓴다. 구현 용어는 DB.md에 둔다 |
| `## 데이터 생명주기`를 `toc`에 포함 | toc는 feature(UI 구분 단위)만 담는다. 데이터 생명주기는 도메인 메타데이터이므로 toc 밖 예약 섹션으로 둔다 |
| 화면명세에 데이터 갱신 주기를 직접 서술 | 갱신 주기는 도메인 속성이므로 feature-spec에 둔다. 화면명세는 features[] 참조로 자동 연결되므로 중복 서술 불필요 |

## toc 변경 시 영향 추적 (필수 절차)

`toc` 항목을 **삭제하거나 id를 변경**할 때는 반드시 다음 grep을 수행하여 모든 참조처를 함께 갱신한다:

```bash
# 1. featureId 전체 검색 (구조적 참조 — area_*.md, wireframe.html)
grep -rn "{DOMAIN}__{TOC_ID}" docs/

# 2. 기능 한글명·UI 요소 검색 (산문 참조 — INDEX, ROUTING, screen.md 본문)
grep -rn "{기능 한글명}\|{관련 UI 요소}" docs/
```

발견되는 곳은 모두 같은 커밋에서 함께 갱신한다. **기능명세만 수정하고 grep을 생략하면 SSOT가 깨지고, 다른 문서들이 폐기된 기능을 계속 광고한다.** lint(`/lint-docs`)가 `area_*.md` features·와이어프레임 `data-feature`의 구조적 참조는 잡지만, 산문 속 잔재(요약 표·쿼리 파라미터 표·인수조건 등)는 사람의 grep만 잡는다.
