# 중단된 /bp:plan 재개

`/bp:plan` 은 인터뷰·명세 작성 중 언제든 끊길 수 있다 (기획자가 창을 닫거나 "나중에" 라고 하거나 세션 종료). 재호출 시 intake.md 상태로 재개.

## intake.md status 값

| status | 의미 | 재호출 시 진입점 |
|---|---|---|
| `drafting` | 요구사항 파싱 직후, 인터뷰 시작 전 | 인터뷰 (첫 질문) |
| `interviewing` | 인터뷰 중, 빈 슬롯 남음 | 인터뷰 (빈 슬롯부터) |
| `ready` | 모든 슬롯 찼음, 작업 트리 예고 전 | 통합 확인 게이트 |
| `awaiting_decision` | planner 가 수동 결정 대기 중 | α 재진입 분기 |
| `needs_attention` | 루프 한계 초과 / 자동수정 연쇄 실패 | 진단 복기 |
| `done` | 명세 작성 완료 | "이미 끝났어요. 수정하려면?" 질문 |

## 재호출 시 첫 턴 흐름 (오케스트레이터)

`/bp:plan {경로}` 호출 받으면:

1. 대상 화면 폴더 판별 (요구사항 md 에서 target, 또는 인자에서 직접)
2. 해당 폴더에 `intake.md` 존재 확인
3. 존재하면 frontmatter `status` 읽기
4. status 별 분기 (아래 참조)

## status 별 복기 톤

### `drafting`

거의 시작 안 된 상태. 기획자는 이전 진행 기억 약할 것.

```
어제 시작해뒀던 {화면명} 있네요. 요구사항은 정리했고 인터뷰는 아직이에요.
이어서 할까요, 처음부터 다시 보실래요?
```

- "이어서" → `interviewing` 으로 전이, 빈 슬롯부터 질문
- "처음부터" → intake.md 삭제 여부 물은 뒤 `drafting` 재생성

### `interviewing`

중간쯤 멈춤. 지금까지 답한 내용을 짧게 복기해준다.

```
어제까지 {화면명} 정리하다 멈추셨어요. 지금까지 이렇게 정리됐어요:

• 상품 상세, 구매자용, PC/모바일
• 필수: 상품 정보 + 옵션 선택
• 이미지 클릭 → 확대

아직 안 정한 건: 리뷰 표시할지, Q&A 탭 여부, 예외 케이스 몇 개.

이어서 볼까요? (바꾸고 싶은 부분 있으면 알려주세요)
```

복기 규칙:
- 채워진 슬롯을 3~5줄로 bullet 요약 (슬롯 이름 숨김)
- 빈 슬롯을 "아직 안 정한 건" 으로 자연어 나열 (슬롯 이름 숨김)
- 기획자가 "그거 바꿔야 해" 하면 해당 슬롯부터 인터뷰 재개

### `ready`

인터뷰 완료, 작업 트리 예고 직전에 중단.

```
{화면명} 인터뷰는 다 끝났고 이제 명세 만들면 되는 상태예요.
바로 만들러 갈까요? 내용 한 번 더 보고 싶으면 말씀해 주세요.
```

- "만들자" → [confirm-gates.md](confirm-gates.md) 의 통합 확인 게이트 (작업 트리 예고) 부터 재개
- "다시 보자" → intake 요약 표시 → 수정 반영 or OK 후 진행

### `awaiting_decision` ⭐

planner 가 수동 결정 대기 중. 가장 복잡한 복기.

1. intake.md `## _pending_decisions` 의 최신 round 읽기
2. 각 항목의 `decision: _pending_` 인 것들 모아서 기획자 facing 질문 재구성 ([planner-ux.md](planner-ux.md) 번역 가이드)
3. 메시지:

```
{화면명} 명세 작성하다가 몇 개 질문드릴 게 있어서 대기 중이었어요:

{각 pending decision 을 자연어 질문으로}

답 주시면 이어서 만들게요. "나중에" 하시면 다시 대기할게요.
```

기획자 답변 수집 → `## _pending_decisions` 에 `decision:` 채움 → **새 Task 로 planner 재호출** (세션이 이미 끝났으므로 SendMessage 가 아닌 새 Task). prompt 에 "intake.md `## _pending_decisions` round {N} 의 decision 필드 반영하고 해당 파일 Edit 후 반송" 지시를 포함하면 planner 가 α-5 로직으로 진입해 결정을 반영 → 오케스트레이터가 이어서 수렴 루프 재개.

(같은 `/bp:plan` 호출 안에서 α 가 발생한 경우는 SendMessage 로 기존 planner 세션 재진입이 정상 경로 — `convergence-loop.md` α-4 참조. resume 시나리오는 세션 종료 후 재호출이므로 새 Task 가 유일 옵션.)

### `needs_attention`

루프 한계 초과 / 연쇄 실패.

```
지난번 만들다가 세 번 시도했는데 몇 가지가 계속 꼬였어요. 같이 보실까요?

걸린 항목:
- {자연어 번역된 위반}

어떻게 풀면 좋을지 같이 얘기해보면:
- 옵션 1: {사용자가 선택할 수 있는 방향}
- 옵션 2: ...
```

기획자와 대화 후 방향 결정 → 결정을 `## _pending_decisions` 에 반영 → planner 재진입 (round 1 재시작).

### `done`

이미 완료된 상태.

```
{화면명} 은 이미 완료돼 있어요.

- docs/screens/.../screen.md (+ 영역/시트/다이얼로그 파일들)
- 관련 기능명세: ...

어떻게 도와드릴까요?
- 수정할 부분이 있다 → 어느 부분?
- 와이어프레임 그리자 → /bp:wireframe {경로}
- 다른 화면 만들자 → /bp:plan 새 요구사항으로
```

## 기존 intake.md 와 다른 요구사항 md 로 재호출

예: 어제 `/bp:plan req-A.md` 로 시작, 오늘 `/bp:plan req-B.md` 로 호출.

intake.md `source` 필드와 새 인자를 비교:

- **다른 경로 + 같은 화면 폴더 타깃** → "요구사항이 바뀌었는데 이 화면 intake 를 업데이트할까요, 새 화면으로 갈까요?"
- **다른 경로 + 다른 화면 폴더** → 전혀 다른 작업. 새 intake.md 생성

## 재개 패턴이 꼭 지켜야 할 것

1. **기술 상태명 노출 금지** — "`awaiting_decision` 상태에서 복귀" ❌. "아직 답 안 준 질문이 있었죠" ⭕
2. **복기는 요약 + 열린 질문** — 기획자가 그대로 이어갈 수도, 수정할 수도, 전면 재검토할 수도 있게
3. **과거 결정 존중** — 이미 채워진 슬롯을 다시 묻지 말 것. 기획자가 명시적으로 "거기 바꾸고 싶어" 해야 재질문
4. **이력 보존** — `_pending_decisions` 과거 round 들은 지우지 않고 쌓는다 (resolved: true 마크). 추후 디버깅·회고용

## 코너 케이스

- **intake.md 는 있는데 본문이 비어 있음** → 드문 경우. 파일 삭제 여부 확인 후 `drafting` 재생성
- **frontmatter status 값이 알려지지 않은 값** → 스킴 변경 가능성. 사용자에게 확인: "status 가 'X' 라고 돼 있는데 이건 없는 값이에요. 어떻게 처리할까요?"
- **intake.md 있고 산출물(screen.md 등) 도 있는데 status 가 `done` 이 아님** → 과거 실행이 마무리 안 됨. 산출물 기준으로 재검토 제안
