# 대화형 인터뷰 전략

오케스트레이터가 `/bp:plan` 에서 기획자와 인터뷰할 때 지키는 패턴. 목적: 기획자가 "심문 받는다" 가 아니라 "동료와 정리한다" 고 느끼게.

## 기본 원칙

### 1. 슬롯이 아니라 빈칸을 본다

`intake` 스킬의 9개 슬롯은 **내부 구조**다. 기획자에게는 보이면 안 된다.

- ❌ "'보여야 할 정보' 슬롯이 비었어요. 다음 3개 질문 드릴게요"
- ⭕ "이 화면에 뭐가 보여야 하는지 조금 더 얘기해 주세요. 리뷰는 있나요? 추천 상품 영역은요?"

### 2. 요구사항에서 뽑을 수 있는 건 먼저 다 뽑는다

입력 요구사항 md 가 있으면, 9개 슬롯을 먼저 **자동 추출** → 채워진 내용을 기획자에게 "이렇게 이해했어요" 로 보여주고 → **진짜 공백만** 질문. 확인 + 질문을 한 문장으로 묶어 턴 수 줄임.

```
요구사항 읽어봤어요. 이렇게 이해했어요:

• 상품 상세, 구매자용, 모바일 우선
• 상품 정보 + 옵션 + 장바구니 (옵션 다 고르면 담기 활성)
• 이미지 클릭 → 확대

맞아요? 그리고 리뷰·Q&A·추천 상품 영역은 있어요 없어요?
```

**왜 이 형태**:
- bullet 3줄 + 묶음 질문 1줄 = 총 5줄 메시지 (기획자가 한눈에 스캔)
- 선택지 세부 ("평점만 / 개별 / 안 함") 는 질문에서 빼고 **기획자가 "리뷰 있어요" 라고 답한 뒤** 후속 1턴에서 파고듦. 한 번에 3축 × 3선택지를 다 묻지 않는다
- "맞아요?" 앞부분과 "영역은 있어요 없어요?" 뒷부분을 **하나의 자연스러운 질문 블록** 으로 — "확인 겸 몇 가지만 더:" 같은 형식적 연결어 제거

자연어 인자나 빈 인자면 처음부터 대화로 채움. 그래도 "한 답변으로 여러 슬롯" 흡수는 유지.

### 3. 한 답변으로 여러 슬롯 흡수

기획자가 "그냥 상세페이지, 이미지·옵션·리뷰 정도 있고 PC/모바일 다" 라고 답하면 **한 답변에서 슬롯 3~4개** 를 채워넣는다. 이후 추가 질문은 진짜 남은 공백만.

### 4. 질문은 2~3개씩, 의미 묶음으로

- ❌ 9개 한꺼번에
- ❌ 1개씩 9턴
- ⭕ "정보 표시 관련" 묶음에서 2~3개, "동작 관련" 묶음에서 2~3개

### 5. 진행 상황은 자연어로

- ❌ "(4/9 슬롯 완료)"
- ⭕ "이제 동작하고 예외 케이스만 정리하면 돼요"
- ⭕ "거의 다 됐어요"

### 6. 추측보다 확인

불명확할 때는 추측하지 말고 짧게 확인:

- ❌ "리뷰는 평점만 표시하는 걸로 정리했습니다"
- ⭕ "리뷰는 평점 + 개수만 요약해서 보여드릴까요, 아니면 개별 리뷰까지 같이?"

## 좋은 질문 / 나쁜 질문

### 좋은 질문
- 구체적 옵션을 제시 ("A / B / 둘 다 / 안 함")
- 예시 동반 ("예: 평점 4.5 (128) 같은 요약만")
- 한 맥락에 묶임 (정보 vs 동작 vs 상태)

### 나쁜 질문
- 기획자가 답하기 어려운 추상 질문 ("유저스토리 뭐예요?", "이 화면 purpose 가?")
- 슬롯 명칭 노출 ("동작·인터랙션 슬롯 내용은?")
- 한 질문에 여러 축 ("리뷰 표시 여부와 옵션 선택 UX 와 예외 케이스는?")

## 슬롯 커버리지 체크 (내부 용도)

오케스트레이터는 인터뷰 중 **내부적으로** 9개 슬롯 커버를 추적한다 (기획자에게 안 보임). `intake` 스킬의 슬롯 정의:

1. 무엇을·누가·왜
2. 유저스토리 (Must/Should/Could)
3. 보여야 할 정보
4. 동작·인터랙션
5. 상태 (로딩·빈·에러)
6. 예외 케이스 (API 실패·권한 등)
7. viewport
8. 시트·다이얼로그
9. 진입 경로 · 참고·메모

각 슬롯이 "기획자 의도로 충분히 채워졌는가" 를 판단. 요구사항에서 명확히 추출됐거나 기획자 답변으로 확정된 슬롯만 채움. 애매한 건 한 번 더 확인.

## 답변 역기록 타이밍

기획자 답변 받을 때마다 즉시 `intake.md` 에 해당 섹션 내용을 업데이트. 매 턴 Edit. 기다렸다가 몰아서 쓰지 않음 — 중간에 세션 끊겨도 진행분 유지.

`status` 전이:
- 파일 막 생성 → `drafting`
- 첫 질문 던진 직후 → `interviewing`
- 9개 슬롯 의미 있게 채워짐 → `ready`

`status` 변화는 파일 안에서만. 기획자 화면에 노출 X.

## 분기: 기존 intake.md 가 있을 때

같은 화면 폴더에 이미 `intake.md` 존재하면 덮어쓰지 않는다. [resume.md](resume.md) 의 복기 플로우로 진입.

## 분기: 여러 화면 후보

요구사항에 2개 이상의 화면이 섞여 있으면, 첫 턴에 기획자에게 가볍게 확인:

```
요구사항에서 화면 몇 개가 보이는데, 같이 가실래요 따로 가실래요?
  - 상품 상세
  - 장바구니
  - 결제

한 번에 1개 씩 진행하는 게 깔끔해요 (지금 /bp:plan 한 번 = 1 화면).
어디서부터 시작할까요?
```

복수 화면을 동시에 굴리지 않는 이유: 인터뷰 맥락이 뒤섞여 답변 품질이 떨어짐.

## Delegation 전 마지막 턴

9개 슬롯 모두 기획자 의도가 잡혔다고 판단되면 → [confirm-gates.md](confirm-gates.md) 의 "통합 확인" 게이트로 넘어감. intake 확정과 작업 트리 예고를 한 턴에 자연스럽게 제시.

## 실패 모드 회피

- **무한 인터뷰** — 기획자가 "몰라, 니가 정해" 라고 반복하면 3턴 안에 "그럼 디폴트로 X 로 갈게요. 나중에 바꿀 수 있어요" 로 매듭
- **과도한 확인** — 이미 명확한 요구사항을 재확인하지 않는다. 애매한 것만 물음
- **슬롯 강박** — 기획자가 "9번(진입경로)은 나중에 정할게" 라고 하면 비운 채로 넘어가도 됨 (단, `intake.md` 본문에 "TBD — 나중에 정함" 으로 기록)
