## 핵심 결정 사항 (Key Decisions)

- **폴더 기반 → 플랫 파일**: feature는 도메인 단위 .md 파일(AUTH.md, PAYMENT.md), screen은 통합 문서(Screen + Requirement + UserStory 3섹션)
- **frontmatter TOC가 단일 소스**: 기능 계층은 TOC에서 정의, 본문 헤딩과 1:1 일치 필수. 최대 4단계(domain → L1 → L2 → L3)
- **featureId는 TOC 파생**: `AUTH__LOGIN__EMAIL_LOGIN` (도메인 + TOC id를 `__`로 연결)
- **elementId는 짧은 키**: `data-el="EMAIL"` — 부모 `data-feature` 스코프 내 유니크. 플랫폼이 `closest('[data-feature]')` + `data-el`로 조합
- **DOM은 중첩, 호버는 CSS-only**: `[data-feature]:hover:not(:has([data-feature]:hover))` — 가장 깊은 feature만 강조, JS 불필요
- **에이전트 = 판단/실행, 스킬 = 포맷 지식**: 공식 문서상 skill과 agent는 독립 병렬 컴포넌트. Claude가 맥락에 따라 자동 선택
- **planning-workflow는 비공식 하네스 패턴**: 공식 문서에 "스킬이 에이전트를 위임" 패턴은 없음. 에이전트로 전환하거나 Claude 자연 라우팅에 맡기는 것이 공식에 가까움
- **reviewer는 hook agent로**: `type: "agent"` hook은 Read/Grep/Glob만 가능(Write 불가) — 리뷰 판정 전용으로 적합

## 시도와 결과 (Attempts & Results)

### 성공한 것
- CSS-only 호버 시뮬레이션으로 4-depth 중첩 문제 해결 확인 (`_simulation-css-only.html`)
- 에이전트/스킬 분리 패턴이 깔끔하게 작동 — 각 파일이 200줄 이하로 유지됨
- CLAUDE.md를 폴더별로 분리하여 on-demand 로드 패턴 적용

### 실패/재고한 것
- 처음에 `data-el`에 full path를 넣으려 했으나 (`AUTH__LOGIN__EMAIL:EMAIL`) → 짧은 키가 더 적절하다고 결론
- planning-workflow를 스킬로 만들었으나 → 공식 패턴이 아님을 확인. 에이전트로 전환하거나 구조 재검토 필요

## 주의사항 (Caveats)

- **공식 플러그인 패턴 vs 현재 구조**: planning-workflow가 "스킬이 에이전트를 위임"하는 비공식 패턴으로 작성됨. 공식은 skill/agent가 독립 병렬 컴포넌트. 다음 단계에서 방향 결정 필요
- **기존 docs/ 데이터는 테스트용**: 전부 삭제해도 됨. 새 구조로 처음부터 다시 만들 예정
- **guides/ 일부 폐기**: FEATURE-FOLDER-SPEC.md, ARCHITECTURE-DECISIONS.md는 새 구조로 대체됨. SKILL-POLICY-SUMMARY.md, SKILLS-SPEC-REFERENCE.md는 검토 후 판단
- **references/ 예제 미갱신**: EXAMPLE.md, PRODUCT-LIST-EXAMPLE.md, REVIEW-TABS.md가 아직 구 구조(data-feature-id, FEATURE_ 접두사)임. 새 구조에 맞게 업데이트 필요
- **hook agent 제한**: `type: "agent"` hook은 Read/Grep/Glob만 가능. reviewer가 직접 파일을 수정할 수 없음 — 판정만 반환하고 메인 에이전트가 수정
