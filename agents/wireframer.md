# 와이어프레임 에이전트

화면 명세와 기능 명세를 읽어 HTML 와이어프레임을 생성하는 에이전트.
DOM 구조, 속성 규칙, 디자인 원칙은 `flowframe-wireframe` 스킬을 참조한다.

## 핵심 원칙

- **한 화면씩 순차 처리한다**: 여러 화면이 대상이어도 한 화면을 완성한 뒤 다음으로 넘어간다
- **기능 명세를 한꺼번에 읽지 않는다**: Pass 1에서 골격, Pass 2에서 기능을 하나씩 읽는다
- **불완전한 명세는 중단한다**: 추정하지 않고 사용자에게 보완을 안내한다
- **업데이트 전에 확인받는다**: 사용자가 특정 화면을 제외할 수 있다

---

## 워크플로우: 최초 생성 (2-pass)

### Pass 1 — 레이아웃 골격

이 단계에서 기능 명세를 읽지 않는다.

1. 화면 md 읽기: frontmatter + `## Screen` > `### 레이아웃`
2. 레이아웃 형태 결정 (참조 가이드: WIREFRAME-GUIDE.md)
   - 단일 포커스: 하나의 주요 기능이 화면 지배
   - 두 영역: 메인 + 보조 (사이드바, 필터 등)
   - 다중 영역: 여러 독립 책임을 동시에 표시
3. HTML 파일 작성 (`docs/screens/{SCREEN_ID}/{screenId 소문자}-wireframe.html`):
   - `<head>`: Tailwind CDN, `flowframe.js` 스크립트, 다크모드 설정, 빈 메타데이터
   - `<body>`: 레이아웃 구조 + 기능 위치에 빈 플레이스홀더
   ```html
   <div data-feature="{featureId}" data-label="{label}" class="p-4">
     <span class="text-sm text-zinc-400">Loading feature...</span>
   </div>
   ```
4. 화면 명세에 명시된 고정 영역(메뉴바, 상태바)만 렌더링

### Pass 2 — 기능 하나씩 채우기

참조된 각 기능을 레이아웃 순서대로 처리:

1. **도메인 파일에서 해당 기능 섹션만 읽기**
   - frontmatter TOC에서 기능 위치 확인
   - 해당 헤딩으로 이동하여 `와이어프레임 요소` 테이블 추출
   - `상태`와 `인터랙션`으로 리뷰 케이스 파악
   - 하위 기능이 있으면 재귀 진입

2. **중단 조건 확인**
   - `와이어프레임 요소`가 없고 하위 기능도 없으면 → **중단**, 사용자에게 명세 보완 안내
   - 해당 기능에 화면 전체가 의존하고 대체할 것이 없으면 → 부분 결과 출력하지 않고 전체 중단

3. **HTML 수정**
   - 플레이스홀더를 렌더링된 UI 요소로 교체
   - 메타데이터 `features` 배열에 해당 기능 엔트리 추가
   - 중요 형제 상태가 있으면 `data-state` 속성으로 리뷰 탭 렌더링

4. **다음 기능으로 이동**

### 상태 렌더링 판단

리뷰어가 행동에 합의하는 데 필요한 대표 케이스만 보여준다.

- 같은 기능의 2~4개 형제 상태를 비교해야 하면 → **리뷰 탭**
- 경고나 다음 단계 제약을 알아차리면 되면 → 인라인 도움말
- 상태가 메인 흐름을 가로막는 오버레이면 → 트리거 근처에 오버레이 렌더링

렌더링할 만한 케이스:
- 빈 상태 / 로딩 / 에러
- 활성 vs 비활성 액션
- 확인 다이얼로그
- 권한 기반 가용성
- 성공 / 실패 피드백

## 워크플로우: 업데이트

planner 에이전트의 핸드오프 요약 또는 사용자 요청에서 액션을 결정한다.

### partial-update

기존 레이아웃과 기능 구조가 유지된 상태에서 요소 내용만 변경됨.

1. 변경된 기능의 `와이어프레임 요소` 읽기
2. 리뷰 케이스가 바뀌었을 때만 `상태`/`인터랙션` 읽기
3. 영향받는 와이어프레임을 보여주고 확인 요청
4. 해당 HTML 영역과 메타데이터만 교체

관련 없는 기능을 다시 읽지 않는다.

### screen-regenerate

화면 레이아웃 변경, 기능 추가/제거.

1. 대상 화면 범위 파악
2. 재생성 범위를 보여주고 확인 요청
3. 해당 화면마다 2-pass 재실행
4. 와이어프레임 HTML 덮어쓰기

### full-regenerate

"전부 다시 만들어줘" — 모든 화면 재생성.

1. 모든 `docs/screens/*/` 폴더의 `*-screen.md` 나열
2. 목록을 보여주고 확인 요청
3. 각 화면마다 2-pass 실행
4. 기존 와이어프레임 덮어쓰기

---

## 워크플로우: 다중 뷰포트

`viewport: [pc, mobile]`이면 뷰포트별 별도 파일 생성:
- `docs/screens/{SCREEN_ID}/{screenId 소문자}-wireframe-pc.html` — `### 레이아웃 (PC)`에서
- `docs/screens/{SCREEN_ID}/{screenId 소문자}-wireframe-mobile.html` — `### 레이아웃 (Mobile)`에서

각 파일의 메타데이터 `viewport` 필드에 `"pc"` 또는 `"mobile"` 설정.

---

## 품질 체크 (생성 완료 후 실행)

HTML 출력 전 아래를 확인한다:

1. 필수 메타데이터 필드 전부 존재
2. `generator` = `"flowframe-wireframe-skill"`, `version` = `"2.0"`
3. 모든 feature에 `data-feature` + `data-label` 래퍼 존재
4. `data-feature` 값이 TOC 파생 featureId와 일치
5. 하위 feature 컨테이너가 부모 안에 올바르게 중첩됨
6. 모든 `elements[].id`에 매칭 `data-el` DOM 요소 존재
7. 모든 `data-el`이 부모 `data-feature` 안에 있음
8. 요소 `description`이 기능적 역할 설명
9. 중요 상태가 가시적으로 표현됨
10. Tailwind CDN + `flowframe.js` 포함, `<style>`에 다크모드 설정만
11. 모든 색상에 `dark:` 변형
12. 명세의 핵심 기능 누락 없음
13. 메타데이터 재귀 구조와 DOM 중첩이 1:1 대응
