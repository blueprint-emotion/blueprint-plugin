# 자동 수정 분류 기준 (plan 산출물)

reviewer 가 feature/screen 산출물의 위반을 "자동 수정 가능" vs "자동 수정 불가" 로 분류하는 기준. planner 는 이 분류에 따라 행동:

- **자동 수정 가능** → planner 가 즉시 Edit (기획자 컨펌 X)
- **자동 수정 불가** → [convergence-loop.md](convergence-loop.md) 의 α 프로토콜로 기획자 결정 수집

## 분류 원칙

### 자동 수정 가능 (safe-to-apply)

**모두** 만족해야 함:

1. **결정적 교정** — 입력이 같으면 항상 같은 수정. 해석 여지 없음
2. **단일 파일 완결** — 한 파일 편집으로 끝. 다른 파일 연쇄 수정 불필요
3. **도메인 의사결정 불필요** — 기획자 판단 개입 없음. 규약·형식 준수 수준
4. **역기능 부작용 없음** — 이 수정이 다른 SSOT·링크·feature 관계를 깨지 않음

### 자동 수정 불가 (needs-human-decision)

**하나라도** 해당:

1. **도메인 의사결정 필요** — featureId 신설·폐기, 도메인 경계 판단, rule 의미 변경
2. **복수 파일 간 구조 변경** — 여러 문서를 일관되게 수정해야 하며 방향성이 여러 가지
3. **해석 여지 있음** — "권장" 이 있어도 어느 대안을 택할지 기획 의도에 따름
4. **기존 기획 의도 변경** — 이미 결정된 요구를 뒤엎는 수정

## Plan 산출물 특화 예시

### 자동 수정 가능

- feature spec 에서 **rule 의 표현만** 개선 (의미 보존, 문장 정리)
- **복붙된 rule** 을 기능명세 링크로 대체 (원본 링크가 명확할 때)
- 깨진 **상대 경로** 정정 (정답이 파일 시스템에서 확인 가능)
- **frontmatter 누락 필드** 추가 (규약대로 기본값 주입: `type`, 접두사 포함 `title` 등)
- toc 에서 **폐기된 featureId 참조** 제거 (참조처가 해당 featureId 를 쓰지 않는 경우에 한함)
- 산문 표의 **폐기된 기능명** 제거 (정의가 이미 없는데 요약에만 남은 경우)
- 비고·설명 구간의 **다른 화면 구현 세부 참조** 제거 (SSOT 경계 위반 문장 한 구절 제거)
- elements 에 **기능명세 속성명 직접 노출** 된 것을 featureId 참조 + 표현 패턴으로 치환
- title 접두사 누락 보완 (`screen_로그인` → `화면_로그인`)

### 자동 수정 불가

- **featureId 자체 신설·삭제** 결정
- 기능명세의 **rule 의미 변경** (표현 정리가 아니라 의미 다른 것으로)
- 영역 `features[]` 에 **무엇을 넣을지** (UI 구분 단위 판단)
- **인수조건의 의미 변경**
- **시트·다이얼로그 신설·폐기** 결정
- **페이지 1개 → 여러 페이지 분리** 여부
- 화면 간 **구조 재배치** (페이지↔시트 변환 등)
- **권한 정책 변경** (누구에게 노출할지)
- 유저스토리의 **Must/Should/Could 우선순위 변경** (릴리스 범위 영향)

## 경계 케이스

### "권장이 명확한데 여러 해석 가능"

예: reviewer 가 "A 섹션을 B 로 이동" 을 권장했지만, "A 유지 + 주석만 추가" 도 유효한 해결.

→ **보수적으로 "자동 수정 불가"**. 해석 여지가 있으면 기획자 컨펌.

### "단일 파일이지만 의미 변경"

예: "이 rule 을 삭제하면 규약 위반 없음" 이라는 reviewer 제안. 기술적으로는 한 파일 교정이지만, rule 삭제는 도메인 의사결정.

→ **"자동 수정 불가"**. "단일 파일" 은 필요조건이지 충분조건 아님.

### "규약 위반이지만 수정 방향 2가지"

예: `screen.md` 의 `features[]` 가 빈 배열인데, 어떤 featureId 를 추가해야 할지 파일만 봐서는 모름.

→ **"자동 수정 불가"**. 어떤 feature 가 맞는지는 기획 맥락.

### "복사된 rule — 링크 대체 or 삭제"

예: area 에 rule 이 복붙되어 있음. 그대로 둔 채 링크 덧붙일 수도, 원본 삭제 후 링크만 둘 수도 있음.

→ 복붙된 rule 의 **원본이 명확** 하면 "자동 수정 가능" (원본 링크로 대체). 원본이 불명확하거나 rule 자체가 원본일 수도 있으면 "자동 수정 불가".

## reviewer 의 분류 포맷

리포트 각 위반 항목에 레이블:

```
- [SSOT] area_option.md `## 인수조건` 에 PRODUCT.md rule 복붙
  - 위치: area_option.md:45
  - 권장: rule 제거 후 [PRODUCT.md#OPTION](...) 링크로 대체
  - **자동 수정 가능** (이유: 원본 PRODUCT.md#OPTION 명확. 복붙 vs 링크는 규약 위반이 정답)
```

또는:

```
- [SSOT] area_option.md 에 새 rule 존재 — PRODUCT 에 없음
  - 위치: area_option.md:45
  - 권장: 이 rule 이 영역 고유인지 PRODUCT 도메인 규칙인지 판단
  - **자동 수정 불가** (이유: 도메인 경계 판단. 영역 유지 vs PRODUCT 승격)
```

이유까지 명시하면 planner 가 경계 케이스에서 판단하기 쉬움.

## planner 의 적용 가이드

### 자동 수정 적용 (즉시 Edit)
- reviewer "권장" 구문을 그대로 반영
- Edit 의 `old_string`/`new_string` 정밀하게
- 한 위반당 한 Edit
- 다른 라인·다른 파일 건드리지 말 것

### 자동 수정 불가 → α 프로토콜
- 기획자 응답 수집 전 해당 위반 수정 금지
- intake.md `## _pending_decisions` 에 payload 기록
- 기획자 결정 도착하면 "자동 수정 가능" 으로 격상 → 다음 라운드에서 적용

## 루프 상호작용

자동 수정 가능 위반만 있으면 기획자 컨펌 0회로 수렴 (빠른 경로). 수동 결정 섞여 있으면 α 프로토콜 1회 이상 개입. 기획자가 "나중에" 선택 시 그 시점의 나머지 위반은 `awaiting_decision` 으로 남김.
